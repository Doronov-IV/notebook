Of starting up

> cd "username/foldernme" // to get to the folder you want to clone repos to
> git clone "web_repos_address" // to actually copy your repos

Of making a commit

> git add . // to add changed files to the scene
> git commit -m "message_text" // to actually make a commit (look one line up)
> git push -u origin "branch_name"(master) // (optional) pushes files to source

Of reseting changes

Way one

> git log // to get commit name
> git reset --hard "commit_name" // to actually reset changes
> git push -f origin "branch_name"(master) // to finally cancell changes

Way two

> git log // to get commit name
> git reset --mixed "commit_name" // to actually reset changes
> git clean -fd (or '-f -d' separately) // to clear the untracked changes
> git push -f origin "branch_name"(master) // to finally cancell changes

// 'git merge origin/master' before 'git push' would cancel those actions

Branching Strategy

After you 'git commit' or 'git push', before writing aything:

> git branch "new_branch_name"
> git checkout "new_branch_name"

Only then should you make any actions with your files, commiting as usual.
If the code doesn't run properly:

> git checkout master (main)

And then you can work on as usual.

If the code works as it should:

> git checkout master (main)
> git merge "new_branch_name"
> git push -u origin master (main) // if needed
